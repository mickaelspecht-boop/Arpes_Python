# ARPES Explorer — contexte projet pour Claude Code

## Objectif général

On développe une petite application Python/PyQt6 pour analyser des données ARPES provenant de différents laboratoires et donc de formats différents. Le but n'est pas d'harmoniser les fichiers bruts, mais d'harmoniser la sortie des loaders vers une structure commune, afin que l'interface et les algorithmes travaillent toujours sur les mêmes champs.

Objectif utilisateur :
- charger des Band Maps (BM) et des Fermi Surfaces (FS) ;
- gérer au moins deux familles de données : Solaris/DA30 et CLS/LNLS ;
- pouvoir calibrer EF ;
- convertir correctement les axes en k ;
- visualiser BM, fit MDC, résultats et FS dans des onglets séparés ;
- garder une interface organisée par dossiers pliables/dépliables ;
- permettre l'analyse de FS avec superposition de ZDB théorique et détection automatique du centre Gamma.

## Architecture actuelle souhaitée

Fichiers principaux :

- `arpes_explorer.py`
  Interface graphique PyQt6 principale.

- `arpes_io.py`
  Couche IO commune. C'est le point d'entrée unique pour charger les données ARPES.

- `arpes_fs.py`
  Module séparé pour tout ce qui concerne les surfaces de Fermi : affichage FS, contrôles lattice/ZDB, normalisation, détection Gamma.

- `arpes_plots.py`
  Fonctions historiques d'analyse ARPES, notamment fitting EF, MDC, courbures, etc. L'explorer peut encore s'en servir.

- `bm.py`
  Script batch historique pour Band Maps Solaris/BaNi2As2.

- `fs_batch.py`
  Script batch historique pour Fermi Surfaces Solaris/BaNi2As2.

- `kz_calc.py`
  Script de calcul kz / V0 pour BaNi2As2.

- `CLS.py`
  Ancien loader CLS écrit comme plugin `peaks`. À conserver comme référence, mais l'application principale ne doit pas dépendre de `peaks` pour CLS.

## Principe fondamental du loader

L'application ne doit plus appeler directement `erlab.io.load(...)` depuis l'interface.

Elle doit toujours faire :

```python
from arpes_io import load_arpes

ds = load_arpes(path, work_func=..., ef_offset=..., a_lattice=..., hv=...)
```

`arpes_io.py` choisit ensuite le loader approprié via `detect_format(path)`.

Logique attendue :

```text
chemin fichier/dossier
    -> detect_format(path)
    -> "solaris_da30" ou "cls_txt"
    -> loader correspondant
    -> ARPESData standard
```

## Structure standard `ARPESData`

`arpes_io.py` doit contenir une dataclass proche de :

```python
@dataclass
class ARPESData:
    data: np.ndarray
    energy: np.ndarray
    kx: Optional[np.ndarray] = None
    ky: Optional[np.ndarray] = None
    hv: Optional[float] = None
    path: Optional[Path] = None
    source_format: str = "unknown"
    metadata: dict[str, Any] = field(default_factory=dict)
    xarray: Any = None
```

Convention pour BM 2D :

```text
data.shape = (n_k, n_E)
kx.shape   = (n_k,)
energy.shape = (n_E,)
```

Convention pour FS 3D :

```text
metadata["fs_data"]   = volume FS, idéalement (n_ky, n_kx, n_E)
metadata["fs_kx"]     = axe kx
metadata["fs_ky"]     = axe ky
metadata["fs_energy"] = axe énergie
metadata["fs_kind"]   = "kxky"
```

L'ancien code de l'interface attend encore parfois un dictionnaire legacy. `ARPESData.as_legacy_bandmap_dict()` doit donc retourner :

```python
{
    "kpar": self.kx,
    "ev_arr": self.energy,
    "data": self.data,
    "hv": self.hv,
    "path": str(self.path),
    "source_format": self.source_format,
    "metadata": self.metadata,
}
```

## Loaders supportés

### Solaris / DA30

Formats : `.ibw`, `.pxt`, `.zip`.

Détection : extension dans `{'.ibw', '.pxt', '.zip'}`.

Loader : `load_solaris_da30_bandmap(...)`.

Comportement attendu :
- utiliser `erlab.io` ;
- appeler `erlab.io.set_loader("da30")` si possible ;
- lire `hv` dans les attributs du fichier ;
- calculer l'énergie relative :

```python
EF_kin = hv - work_func
E_minus_EF = Ekin - EF_kin + ef_offset
```

- appliquer `da_be.kspace.convert()` ;
- convertir `kx` et `ky` en unités `pi/a` :

```python
kx_pi_a = kx_Ainv * a_lattice / np.pi
ky_pi_a = ky_Ainv * a_lattice / np.pi
```

- si c'est un volume 3D avec kx/ky/eV, stocker le volume dans `metadata['fs_data']`, etc. ;
- si c'est une BM 2D, retourner `data` en shape `(nk, nE)`.

### CLS / LNLS txt autonome

Le fichier `CLS.py` original dépend de `peaks`. Il faut garder cette version comme référence seulement.

L'application principale doit charger CLS via `arpes_io.py`, sans importer `peaks`.

Formats CLS :

1. BM CLS :

```text
BM8
BM8_param.txt
```

Le fichier de données n'a pas forcément d'extension. Le param associé est `<nom>_param.txt`.

2. FS CLS :

```text
FS_folder/
    prefix_param.txt
    prefix_Cycle_C_Step_S.txt
```

Cycles = répétitions à moyenner.
Steps = positions de tilt/phi à convertir en axe FS vertical.

Paramètres dans `_param.txt` :

```text
Pass energy: ...
Lens mode: ...
Acquisition mode: ...
Central Energy: ...
Dwell Time: ...
Energy min: ...; Energy delta: ...
Angle min: ...; Angle delta: ...
{ JSON positions manipulateur }
```

Le fichier CLS ne stocke pas toujours `hv`. L'utilisateur doit pouvoir renseigner `CLS hν` manuellement dans l'interface. Cette valeur doit persister dans la session et être réutilisée pour les BM/FS suivantes.

Pour CLS :
- reconstruire l'axe énergie brute cinétique à partir de `Energy min` et `Energy delta` ;
- reconstruire l'axe angle à partir de `Angle min` et `Angle delta` ;
- utiliser `Central Energy` comme référence EF cinétique initiale quand c'est plus fiable pour Ba122 CLS ;
- convertir l'énergie vers `E - EF` avec `ef_offset` ;
- convertir `theta_par` en `kx (pi/a)` ;
- pour FS, convertir `tilt/phi` en `ky (pi/a)`.

Conversion typique :

```python
k = 0.51233 * sqrt(Ekin_EF) * sin(angle_deg) * a_lattice / np.pi
```

Attention : pour CLS/Ba122, dans le notebook V4 l'axe FS utilisait une logique de centrage Gamma et conversion tilt/phi spécifique. Le loader doit éviter de laisser l'axe vertical en degrés dans l'onglet FS.

## Calibration EF

Objectif : placer le bord de Fermi à `E = 0 eV`.

Workflow :

```text
données chargées
-> moyenne sur k pour obtenir une EDC moyenne
-> fit du bord de Fermi
-> mesure du décalage EF
-> correction de ef_offset
-> sauvegarde dans la session
-> reload du fichier avec le nouvel offset
```

Le fit utilise une Fermi-Dirac convoluée par une gaussienne de résolution avec fond simple.

La calibration doit être robuste :
- si EF n'est pas déjà proche de zéro, chercher automatiquement le plus fort gradient de l'EDC et adapter la fenêtre de fit ;
- afficher un dialogue ou message avec :
  - EF trouvé en meV ;
  - fenêtre utilisée ;
  - FWHM résolution ;
  - résidu RMS ;
  - ancien offset ;
  - nouvel offset proposé.

Important : l'offset doit être écrit dans la session AVANT de recharger le fichier. Un bug précédent faisait que `_load_file()` restaurait l'ancien offset et écrasait le nouveau.

Important aussi : l'offset sauvegardé doit être appliqué AVANT le chargement du fichier.

### Raw / EDCnorm

Le mode `Raw` doit vraiment afficher les données brutes.

Le mode `EDCnorm` doit afficher :

```python
edc = np.nanmean(data, axis=0, keepdims=True)
data_norm = data / safe_edc
```

`SecDev` / `Curvature` peuvent utiliser soit le brut soit EDCnorm selon le checkbox, mais `Raw` ne doit jamais être modifié par EDCnorm.

## Mise à jour récente — état de l'app après optimisations et corrections

Cette section résume les changements récents à connaître avant de reprendre le
développement. Elle est écrite pour donner à un autre agent de code le contexte
nécessaire sans relire toute la conversation.

### 1. Référence Gamma globale du dossier

Objectif :
- quand l'utilisateur clique sur `Détecter Γ FS` ou `Auto Γ BM`, le centre
  trouvé doit être conservé et réutilisé pour les autres fichiers du même
  dossier de données.

Contexte :
- auparavant `Détecter Γ FS` mettait bien à jour le panneau FS et écrivait
  `session.gamma_reference`, mais cette référence n'était pas appliquée
  automatiquement aux fichiers chargés ensuite.
- l'utilisateur veut mesurer Γ une fois, puis traiter toutes les BM/FS du
  dossier avec ce même centre de référence.

Modifications :
- dans `arpes_explorer.py`, ajout de helpers :
  - `_stored_gamma_reference()`
  - `_gamma_reference_to_bm_center(ref)`
  - `_apply_stored_gamma_to_current_file(save_entry=False)`
- `_detect_fs_gamma()` ajoute maintenant `source: "fs"` dans
  `session.gamma_reference` et sauvegarde la session.
- `_estimate_gamma_bm()` sauvegarde aussi une `gamma_reference` globale avec
  `source: "bm"`.
- `_load_file()` appelle `_apply_stored_gamma_to_current_file(save_entry=True)`
  après restauration des paramètres du fichier.
- pour une FS, la référence sauvegardée met à jour `centre kx` et `centre ky`
  dans le panneau FS.
- pour une BM, la référence sauvegardée met à jour `center_init`, avec la
  correction de polar déjà prévue dans le code si nécessaire.

Point important :
- ne pas supprimer la correction de polar dans `_gamma_reference_to_bm_center`.
  Elle évite d'appliquer directement un Γ FS à une BM prise à une polar
  différente si `polar_already_applied_to_kx` n'est pas déjà vrai des deux côtés.

### 2. Normalisation FS commune et contrôle des bords

Objectif :
- les FS Solaris et CLS doivent utiliser une logique de normalisation cohérente
  côté affichage.
- les bords `kx/theta` peuvent être très intenses sur CLS et écraser le signal
  utile au centre de la FS.

Contexte :
- l'ancienne normalisation FS corrigeait surtout un profil de flux par slice
  selon l'axe lent `ky/tilt`.
- pour CLS, un profil détecteur selon `kx/theta` peut aussi être fort aux
  bords. Une correction `y+x` a donc été restaurée pour les FS.
- une tentative d'appliquer une correction de flux aux BM a changé leur
  apparence. Cette partie a été annulée : les BM gardent leur comportement
  historique.

Modifications :
- ajout de `arpes_norm.py`, module commun de normalisation.
- fonctions principales dans `arpes_norm.py` :
  - `normalize_bandmap_flux_profile(...)` existe mais n'est actuellement pas
    utilisée par l'affichage BM.
  - `normalize_fs_flux_profiles(...)` normalise un volume FS complet.
  - `fs_flux_profile_factors(...)` calcule les facteurs de correction sans
    copier tout le volume 3D.
  - `apply_fs_flux_factors_to_map(...)` applique ces facteurs sur la carte FS
    2D affichée.
- dans `arpes_fs.py`, `extract_fs_map(...)` :
  - calcule d'abord la carte FS 2D par moyenne autour de EF ;
  - calcule ensuite les facteurs de flux sur le volume brut avec
    `normalize_y=True` et `normalize_x=True` ;
  - applique ces facteurs à la carte 2D affichée.
- cette version est équivalente au fait de normaliser le volume puis de moyenner
  autour de EF, mais évite de recopier un gros volume 3D à chaque redessin.
- ajout dans `FSParams` :
  - `edge_trim_x: float = 0.08`
- ajout dans `FSControlPanel` :
  - contrôle `Masquer bords kx %`, valeur par défaut `8.0`.
- `extract_fs_map(...)` masque les colonnes extrêmes gauche/droite de la carte
  FS avant la normalisation d'affichage robuste. Cela évite que les bords très
  forts imposent l'échelle couleur.

Comportement actuel :
- FS : correction flux `y+x` active si `Normalisation flux par slice` est cochée.
- FS : les bords `kx` sont masqués par défaut à 8 %, réglable de 0 à 20 %.
- BM : `Raw` reste brut.
- BM : `EDCnorm` reste l'ancien comportement `data / mean_k(data)`, sans
  correction de flux additionnelle.

Point important :
- le label `Normalisation flux par slice` est historiquement nommé ainsi, mais
  le tooltip précise désormais que la correction inclut aussi le profil
  détecteur `kx`.
- si l'utilisateur trouve que trop de signal utile est coupé aux bords, réduire
  `Masquer bords kx %` dans l'onglet FS.
- si les bords restent dominants, augmenter ce pourcentage à 10-15 %.

### 3. Optimisation du chargement CLS FS

Objectif :
- accélérer fortement le chargement des FS CLS, qui lisent beaucoup de gros
  fichiers texte `*_Cycle_*_Step_*.txt`.

Contexte :
- le goulot principal n'est pas un calcul NumPy, mais la lecture/parsing de
  dizaines de fichiers texte.
- sur un exemple réel `Ba122_C05_2/FS5`, il y a 77 fichiers texte d'environ
  7 Mo chacun.
- utiliser des bibliothèques C externes n'est pas prioritaire : NumPy utilise
  déjà du C pour les opérations tableaux. Le gain majeur vient du cache et de
  la réduction des lectures répétées.

Modifications dans `arpes_io.py` :
- ajout de `_loadtxt_float32(path)` pour lire directement les fichiers CLS en
  `float32`.
- ajout d'une logique cache CLS FS :
  - `_CLS_CACHE_VERSION = 2`
  - `_cls_fs_cache_path(folder, prefix)`
  - `_cls_fs_signature(param_file, files)`
  - `_load_cls_fs_cache(cache_path, signature)`
  - `_save_cls_fs_cache(cache_path, signature, volume, step_ids)`
  - `_load_cls_fs_volume(folder, prefix)`
- `_load_cls_fs_volume(...)` :
  - groupe les fichiers par `Step`;
  - moyenne les cycles de chaque step ;
  - sauvegarde le volume moyenné dans `.arpes_cache/<prefix>_fs_mean_v2.npz`;
  - relit ce cache si les fichiers source et le `_param.txt` n'ont pas changé.
- le cache est non compressé (`np.savez`, pas `np.savez_compressed`) pour
  privilégier la vitesse de chargement.
- lecture parallèle des fichiers texte avec `ThreadPoolExecutor`, limitée à
  8 workers.

Mesures sur un cas réel :
- dossier testé : `/Users/alexandrespecht/Documents/Stage_M2/Code/Ba122_C05_2/FS5`
- premier chargement avec cache à construire : environ `8.55 s`
- chargement suivant depuis cache : environ `0.28 s`
- la comparaison avec l'ancien algorithme donne :
  - `loader diff = 0.0`
  - `raw mean diff = 0.0`
- donc l'optimisation IO ne change pas les données reconstruites.

Modifications dans `.gitignore` :
- ajout de :

```text
.arpes_cache/
**/.arpes_cache/
```

Point important :
- ne pas versionner les caches `.arpes_cache`, ils peuvent être gros.
- le cache s'invalide si un fichier Cycle/Step ou le `_param.txt` change
  (`size` et `mtime_ns` inclus dans la signature).
- si un comportement de chargement CLS semble suspect, comparer d'abord le
  volume optimisé avec l'ancien algorithme avant de supprimer le cache.

### 4. Régressions rencontrées et corrections

Régression : BM changée visuellement après normalisation flux commune.
- Cause : la correction de flux avait été appliquée aux BM avant `EDCnorm`.
- Correction : BM restaurée à l'ancien comportement.
- État actuel :
  - `apply_edcnorm(data)` dans `arpes_explorer.py` ne prend plus l'énergie et
    fait seulement `data / mean_k(data)`.
  - les appels à `apply_edcnorm(...)` sont revenus à un seul argument.

### 7. Conseils pour la suite

- Ne pas modifier simultanément le loader et la normalisation d'affichage sans
  test comparatif, sinon il devient difficile d'attribuer les régressions.
- Pour déboguer CLS FS :
  1. comparer d'abord le volume brut `fs_data` avec l'ancien algorithme ;
  2. vérifier ensuite la BM moyenne `volume.mean(axis=0)`;
  3. vérifier seulement après la normalisation FS affichée.
- Pour améliorer encore les performances :
  - priorité 1 : éviter les redessins FS inutiles ;
  - priorité 2 : mémoriser la carte FS 2D normalisée pour une combinaison donnée
    de paramètres (`ef_window`, `norm_ref_lo/hi`, lissage) ;
  - priorité 3 : envisager un cache `.npy` ou `np.memmap` si les volumes
    deviennent trop gros pour la RAM.

## Mise à jour 2026-04-29 — mode Igor pour suppression grille

- Le fichier Igor pertinent a été retrouvé :
  `/Users/alexandrespecht/Downloads/ARPES_analyzer/miscellaneous/manual_grid_remove.ipf`.
- Son principe :
  - `FFT_gridrem2D(w, wmask)` calcule magnitude, phase et FFT complexe 2D ;
  - `FFT_mask_remove(wmag, wmask)` remplace les pixels de magnitude où
    `wmask == 0` par la moyenne de la magnitude ;
  - la FFT complexe est reconstruite avec la magnitude modifiée et la phase
    originale ;
  - `IFFT` reconstruit l'image corrigée ;
  - `FFT_gridrem3D` applique cette correction 2D slice par slice.
- Ajout dans l'app d'une méthode interne `Masque Fourier 2D` :
  - méthode interne `fft2mask` dans `arpes_norm.py` ;
  - pour une FS `(ny, nx, E)`, la correction est appliquée slice énergie par
    slice sur les plans `(ny, nx)`, ce qui correspond au comportement Igor ;
  - pour une BM 2D, la correction agit directement sur l'image 2D ;
  - le masque est automatique : il protège le centre de la FFT puis détecte les
    pics anormaux de magnitude via un seuil robuste sur `log(1 + |FFT|)`.
- Correction importante après retour utilisateur : le premier `fft2mask`
  corrigeait les plans `(ky, kx)` à énergie fixe. Pour une grille de détecteur
  Solaris, ce n'est généralement pas le bon plan : Igor corrige les images
  détecteur 2D puis boucle sur la troisième dimension. Dans l'app, une FS est
  standardisée en `(beta/ky, theta/kx, E)`, donc le mode par défaut corrige
  maintenant les plans `theta/kx - E` pour chaque slice `beta/ky`.
- Le statut de correction affiche maintenant :
  - le nombre de pixels FFT masqués ;
  - le nombre de slices touchées pour les volumes FS ;
  - `Δ≈...%`, estimation RMS du changement relatif. Si `Δ≈0%`, la correction
    n'a pratiquement rien modifié et le seuil/plan est probablement mauvais.
