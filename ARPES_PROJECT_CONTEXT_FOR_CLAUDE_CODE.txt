# ARPES Explorer — contexte projet pour Claude Code

## Objectif général

On développe une petite application Python/PyQt6 pour analyser des données ARPES provenant de différents laboratoires et donc de formats différents. Le but n'est pas d'harmoniser les fichiers bruts, mais d'harmoniser la sortie des loaders vers une structure commune, afin que l'interface et les algorithmes travaillent toujours sur les mêmes champs.

Objectif utilisateur :
- charger des Band Maps (BM) et des Fermi Surfaces (FS) ;
- gérer au moins deux familles de données : Solaris/DA30 et CLS/LNLS ;
- pouvoir calibrer EF ;
- convertir correctement les axes en k ;
- visualiser BM, fit MDC, résultats et FS dans des onglets séparés ;
- garder une interface organisée par dossiers pliables/dépliables ;
- permettre l'analyse de FS avec superposition de ZDB théorique et détection automatique du centre Gamma.

## Architecture actuelle souhaitée

Fichiers principaux :

- `arpes_explorer.py`
  Interface graphique PyQt6 principale.

- `arpes_io.py`
  Couche IO commune. C'est le point d'entrée unique pour charger les données ARPES.

- `arpes_fs.py`
  Module séparé pour tout ce qui concerne les surfaces de Fermi : affichage FS, contrôles lattice/ZDB, normalisation, détection Gamma.

- `arpes_plots.py`
  Fonctions historiques d'analyse ARPES, notamment fitting EF, MDC, courbures, etc. L'explorer peut encore s'en servir.

- `bm.py`
  Script batch historique pour Band Maps Solaris/BaNi2As2.

- `fs_batch.py`
  Script batch historique pour Fermi Surfaces Solaris/BaNi2As2.

- `kz_calc.py`
  Script de calcul kz / V0 pour BaNi2As2.

- `CLS.py`
  Ancien loader CLS écrit comme plugin `peaks`. À conserver comme référence, mais l'application principale ne doit pas dépendre de `peaks` pour CLS.

## Principe fondamental du loader

L'application ne doit plus appeler directement `erlab.io.load(...)` depuis l'interface.

Elle doit toujours faire :

```python
from arpes_io import load_arpes

ds = load_arpes(path, work_func=..., ef_offset=..., a_lattice=..., hv=...)
```

`arpes_io.py` choisit ensuite le loader approprié via `detect_format(path)`.

Logique attendue :

```text
chemin fichier/dossier
    -> detect_format(path)
    -> "solaris_da30" ou "cls_txt"
    -> loader correspondant
    -> ARPESData standard
```

## Structure standard `ARPESData`

`arpes_io.py` doit contenir une dataclass proche de :

```python
@dataclass
class ARPESData:
    data: np.ndarray
    energy: np.ndarray
    kx: Optional[np.ndarray] = None
    ky: Optional[np.ndarray] = None
    hv: Optional[float] = None
    path: Optional[Path] = None
    source_format: str = "unknown"
    metadata: dict[str, Any] = field(default_factory=dict)
    xarray: Any = None
```

Convention pour BM 2D :

```text
data.shape = (n_k, n_E)
kx.shape   = (n_k,)
energy.shape = (n_E,)
```

Convention pour FS 3D :

```text
metadata["fs_data"]   = volume FS, idéalement (n_ky, n_kx, n_E)
metadata["fs_kx"]     = axe kx
metadata["fs_ky"]     = axe ky
metadata["fs_energy"] = axe énergie
metadata["fs_kind"]   = "kxky"
```

L'ancien code de l'interface attend encore parfois un dictionnaire legacy. `ARPESData.as_legacy_bandmap_dict()` doit donc retourner :

```python
{
    "kpar": self.kx,
    "ev_arr": self.energy,
    "data": self.data,
    "hv": self.hv,
    "path": str(self.path),
    "source_format": self.source_format,
    "metadata": self.metadata,
}
```

## Loaders supportés

### Solaris / DA30

Formats : `.ibw`, `.pxt`, `.zip`.

Détection : extension dans `{'.ibw', '.pxt', '.zip'}`.

Loader : `load_solaris_da30_bandmap(...)`.

Comportement attendu :
- utiliser `erlab.io` ;
- appeler `erlab.io.set_loader("da30")` si possible ;
- lire `hv` dans les attributs du fichier ;
- calculer l'énergie relative :

```python
EF_kin = hv - work_func
E_minus_EF = Ekin - EF_kin + ef_offset
```

- appliquer `da_be.kspace.convert()` ;
- convertir `kx` et `ky` en unités `pi/a` :

```python
kx_pi_a = kx_Ainv * a_lattice / np.pi
ky_pi_a = ky_Ainv * a_lattice / np.pi
```

- si c'est un volume 3D avec kx/ky/eV, stocker le volume dans `metadata['fs_data']`, etc. ;
- si c'est une BM 2D, retourner `data` en shape `(nk, nE)`.

### CLS / LNLS txt autonome

Le fichier `CLS.py` original dépend de `peaks`. Il faut garder cette version comme référence seulement.

L'application principale doit charger CLS via `arpes_io.py`, sans importer `peaks`.

Formats CLS :

1. BM CLS :

```text
BM8
BM8_param.txt
```

Le fichier de données n'a pas forcément d'extension. Le param associé est `<nom>_param.txt`.

2. FS CLS :

```text
FS_folder/
    prefix_param.txt
    prefix_Cycle_C_Step_S.txt
```

Cycles = répétitions à moyenner.
Steps = positions de tilt/phi à convertir en axe FS vertical.

Paramètres dans `_param.txt` :

```text
Pass energy: ...
Lens mode: ...
Acquisition mode: ...
Central Energy: ...
Dwell Time: ...
Energy min: ...; Energy delta: ...
Angle min: ...; Angle delta: ...
{ JSON positions manipulateur }
```

Le fichier CLS ne stocke pas toujours `hv`. L'utilisateur doit pouvoir renseigner `CLS hν` manuellement dans l'interface. Cette valeur doit persister dans la session et être réutilisée pour les BM/FS suivantes.

Pour CLS :
- reconstruire l'axe énergie brute cinétique à partir de `Energy min` et `Energy delta` ;
- reconstruire l'axe angle à partir de `Angle min` et `Angle delta` ;
- utiliser `Central Energy` comme référence EF cinétique initiale quand c'est plus fiable pour Ba122 CLS ;
- convertir l'énergie vers `E - EF` avec `ef_offset` ;
- convertir `theta_par` en `kx (pi/a)` ;
- pour FS, convertir `tilt/phi` en `ky (pi/a)`.

Conversion typique :

```python
k = 0.51233 * sqrt(Ekin_EF) * sin(angle_deg) * a_lattice / np.pi
```

Attention : pour CLS/Ba122, dans le notebook V4 l'axe FS utilisait une logique de centrage Gamma et conversion tilt/phi spécifique. Le loader doit éviter de laisser l'axe vertical en degrés dans l'onglet FS.

## Calibration EF

Objectif : placer le bord de Fermi à `E = 0 eV`.

Workflow :

```text
données chargées
-> moyenne sur k pour obtenir une EDC moyenne
-> fit du bord de Fermi
-> mesure du décalage EF
-> correction de ef_offset
-> sauvegarde dans la session
-> reload du fichier avec le nouvel offset
```

Le fit utilise une Fermi-Dirac convoluée par une gaussienne de résolution avec fond simple.

La calibration doit être robuste :
- si EF n'est pas déjà proche de zéro, chercher automatiquement le plus fort gradient de l'EDC et adapter la fenêtre de fit ;
- afficher un dialogue ou message avec :
  - EF trouvé en meV ;
  - fenêtre utilisée ;
  - FWHM résolution ;
  - résidu RMS ;
  - ancien offset ;
  - nouvel offset proposé.

Important : l'offset doit être écrit dans la session AVANT de recharger le fichier. Un bug précédent faisait que `_load_file()` restaurait l'ancien offset et écrasait le nouveau.

Important aussi : l'offset sauvegardé doit être appliqué AVANT le chargement du fichier.

### Raw / EDCnorm

Le mode `Raw` doit vraiment afficher les données brutes.

Le mode `EDCnorm` doit afficher :

```python
edc = np.nanmean(data, axis=0, keepdims=True)
data_norm = data / safe_edc
```

`SecDev` / `Curvature` peuvent utiliser soit le brut soit EDCnorm selon le checkbox, mais `Raw` ne doit jamais être modifié par EDCnorm.

## Mise à jour récente — état de l'app après optimisations et corrections

Cette section résume les changements récents à connaître avant de reprendre le
développement. Elle est écrite pour donner à un autre agent de code le contexte
nécessaire sans relire toute la conversation.

### 1. Référence Gamma globale du dossier

Objectif :
- quand l'utilisateur clique sur `Détecter Γ FS` ou `Auto Γ BM`, le centre
  trouvé doit être conservé et réutilisé pour les autres fichiers du même
  dossier de données.

Contexte :
- auparavant `Détecter Γ FS` mettait bien à jour le panneau FS et écrivait
  `session.gamma_reference`, mais cette référence n'était pas appliquée
  automatiquement aux fichiers chargés ensuite.
- l'utilisateur veut mesurer Γ une fois, puis traiter toutes les BM/FS du
  dossier avec ce même centre de référence.

Modifications :
- dans `arpes_explorer.py`, ajout de helpers :
  - `_stored_gamma_reference()`
  - `_gamma_reference_to_bm_center(ref)`
  - `_apply_stored_gamma_to_current_file(save_entry=False)`
- `_detect_fs_gamma()` ajoute maintenant `source: "fs"` dans
  `session.gamma_reference` et sauvegarde la session.
- `_estimate_gamma_bm()` sauvegarde aussi une `gamma_reference` globale avec
  `source: "bm"`.
- `_load_file()` appelle `_apply_stored_gamma_to_current_file(save_entry=True)`
  après restauration des paramètres du fichier.
- pour une FS, la référence sauvegardée met à jour `centre kx` et `centre ky`
  dans le panneau FS.
- pour une BM, la référence sauvegardée met à jour `center_init`, avec la
  correction de polar déjà prévue dans le code si nécessaire.

Point important :
- ne pas supprimer la correction de polar dans `_gamma_reference_to_bm_center`.
  Elle évite d'appliquer directement un Γ FS à une BM prise à une polar
  différente si `polar_already_applied_to_kx` n'est pas déjà vrai des deux côtés.

### 2. Normalisation FS commune et contrôle des bords

Objectif :
- les FS Solaris et CLS doivent utiliser une logique de normalisation cohérente
  côté affichage.
- les bords `kx/theta` peuvent être très intenses sur CLS et écraser le signal
  utile au centre de la FS.

Contexte :
- l'ancienne normalisation FS corrigeait surtout un profil de flux par slice
  selon l'axe lent `ky/tilt`.
- pour CLS, un profil détecteur selon `kx/theta` peut aussi être fort aux
  bords. Une correction `y+x` a donc été restaurée pour les FS.
- une tentative d'appliquer une correction de flux aux BM a changé leur
  apparence. Cette partie a été annulée : les BM gardent leur comportement
  historique.

Modifications :
- ajout de `arpes_norm.py`, module commun de normalisation.
- fonctions principales dans `arpes_norm.py` :
  - `normalize_bandmap_flux_profile(...)` existe mais n'est actuellement pas
    utilisée par l'affichage BM.
  - `normalize_fs_flux_profiles(...)` normalise un volume FS complet.
  - `fs_flux_profile_factors(...)` calcule les facteurs de correction sans
    copier tout le volume 3D.
  - `apply_fs_flux_factors_to_map(...)` applique ces facteurs sur la carte FS
    2D affichée.
- dans `arpes_fs.py`, `extract_fs_map(...)` :
  - calcule d'abord la carte FS 2D par moyenne autour de EF ;
  - calcule ensuite les facteurs de flux sur le volume brut avec
    `normalize_y=True` et `normalize_x=True` ;
  - applique ces facteurs à la carte 2D affichée.
- cette version est équivalente au fait de normaliser le volume puis de moyenner
  autour de EF, mais évite de recopier un gros volume 3D à chaque redessin.
- ajout dans `FSParams` :
  - `edge_trim_x: float = 0.08`
- ajout dans `FSControlPanel` :
  - contrôle `Masquer bords kx %`, valeur par défaut `8.0`.
- `extract_fs_map(...)` masque les colonnes extrêmes gauche/droite de la carte
  FS avant la normalisation d'affichage robuste. Cela évite que les bords très
  forts imposent l'échelle couleur.

Comportement actuel :
- FS : correction flux `y+x` active si `Normalisation flux par slice` est cochée.
- FS : les bords `kx` sont masqués par défaut à 8 %, réglable de 0 à 20 %.
- BM : `Raw` reste brut.
- BM : `EDCnorm` reste l'ancien comportement `data / mean_k(data)`, sans
  correction de flux additionnelle.

Point important :
- le label `Normalisation flux par slice` est historiquement nommé ainsi, mais
  le tooltip précise désormais que la correction inclut aussi le profil
  détecteur `kx`.
- si l'utilisateur trouve que trop de signal utile est coupé aux bords, réduire
  `Masquer bords kx %` dans l'onglet FS.
- si les bords restent dominants, augmenter ce pourcentage à 10-15 %.

### 3. Optimisation du chargement CLS FS

Objectif :
- accélérer fortement le chargement des FS CLS, qui lisent beaucoup de gros
  fichiers texte `*_Cycle_*_Step_*.txt`.

Contexte :
- le goulot principal n'est pas un calcul NumPy, mais la lecture/parsing de
  dizaines de fichiers texte.
- sur un exemple réel `Ba122_C05_2/FS5`, il y a 77 fichiers texte d'environ
  7 Mo chacun.
- utiliser des bibliothèques C externes n'est pas prioritaire : NumPy utilise
  déjà du C pour les opérations tableaux. Le gain majeur vient du cache et de
  la réduction des lectures répétées.

Modifications dans `arpes_io.py` :
- ajout de `_loadtxt_float32(path)` pour lire directement les fichiers CLS en
  `float32`.
- ajout d'une logique cache CLS FS :
  - `_CLS_CACHE_VERSION = 2`
  - `_cls_fs_cache_path(folder, prefix)`
  - `_cls_fs_signature(param_file, files)`
  - `_load_cls_fs_cache(cache_path, signature)`
  - `_save_cls_fs_cache(cache_path, signature, volume, step_ids)`
  - `_load_cls_fs_volume(folder, prefix)`
- `_load_cls_fs_volume(...)` :
  - groupe les fichiers par `Step`;
  - moyenne les cycles de chaque step ;
  - sauvegarde le volume moyenné dans `.arpes_cache/<prefix>_fs_mean_v2.npz`;
  - relit ce cache si les fichiers source et le `_param.txt` n'ont pas changé.
- le cache est non compressé (`np.savez`, pas `np.savez_compressed`) pour
  privilégier la vitesse de chargement.
- lecture parallèle des fichiers texte avec `ThreadPoolExecutor`, limitée à
  8 workers.

Mesures sur un cas réel :
- dossier testé : `/Users/alexandrespecht/Documents/Stage_M2/Code/Ba122_C05_2/FS5`
- premier chargement avec cache à construire : environ `8.55 s`
- chargement suivant depuis cache : environ `0.28 s`
- la comparaison avec l'ancien algorithme donne :
  - `loader diff = 0.0`
  - `raw mean diff = 0.0`
- donc l'optimisation IO ne change pas les données reconstruites.

Modifications dans `.gitignore` :
- ajout de :

```text
.arpes_cache/
**/.arpes_cache/
```

Point important :
- ne pas versionner les caches `.arpes_cache`, ils peuvent être gros.
- le cache s'invalide si un fichier Cycle/Step ou le `_param.txt` change
  (`size` et `mtime_ns` inclus dans la signature).
- si un comportement de chargement CLS semble suspect, comparer d'abord le
  volume optimisé avec l'ancien algorithme avant de supprimer le cache.

### 4. Régressions rencontrées et corrections

Régression : BM changée visuellement après normalisation flux commune.
- Cause : la correction de flux avait été appliquée aux BM avant `EDCnorm`.
- Correction : BM restaurée à l'ancien comportement.
- État actuel :
  - `apply_edcnorm(data)` dans `arpes_explorer.py` ne prend plus l'énergie et
    fait seulement `data / mean_k(data)`.
  - les appels à `apply_edcnorm(...)` sont revenus à un seul argument.

## Mise à jour 2026-04-29 — mode Igor pour suppression grille

- Le fichier Igor pertinent a été retrouvé :
  `/Users/alexandrespecht/Downloads/ARPES_analyzer/miscellaneous/manual_grid_remove.ipf`.
- Son principe :
  - `FFT_gridrem2D(w, wmask)` calcule magnitude, phase et FFT complexe 2D ;
  - `FFT_mask_remove(wmag, wmask)` remplace les pixels de magnitude où
    `wmask == 0` par la moyenne de la magnitude ;
  - la FFT complexe est reconstruite avec la magnitude modifiée et la phase
    originale ;
  - `IFFT` reconstruit l'image corrigée ;
  - `FFT_gridrem3D` applique cette correction 2D slice par slice.
- Ajout dans l'app d'une méthode interne `Masque Fourier 2D` :
  - méthode interne `fft2mask` dans `arpes_norm.py` ;
  - pour une FS `(ny, nx, E)`, la correction est appliquée slice énergie par
    slice sur les plans `(ny, nx)`, ce qui correspond au comportement Igor ;
  - pour une BM 2D, la correction agit directement sur l'image 2D ;
  - le masque est automatique : il protège le centre de la FFT puis détecte les
    pics anormaux de magnitude via un seuil robuste sur `log(1 + |FFT|)`.
- Correction importante après retour utilisateur : le premier `fft2mask`
  corrigeait les plans `(ky, kx)` à énergie fixe. Pour une grille de détecteur
  Solaris, ce n'est généralement pas le bon plan : Igor corrige les images
  détecteur 2D puis boucle sur la troisième dimension. Dans l'app, une FS est
  standardisée en `(beta/ky, theta/kx, E)`, donc le mode par défaut corrige
  maintenant les plans `theta/kx - E` pour chaque slice `beta/ky`.
- Le statut de correction affiche maintenant :
  - le nombre de pixels FFT masqués ;
  - le nombre de slices touchées pour les volumes FS ;
  - `Δ≈...%`, estimation RMS du changement relatif. Si `Δ≈0%`, la correction
    n'a pratiquement rien modifié et le seuil/plan est probablement mauvais.

## TODO Point 6 — Centrage Γ par offsets angulaires (au lieu de translation k)

### Constat

Aujourd'hui, le centrage Γ se fait par translation des axes `k` déjà
convertis :
- `find_gamma_mdc` ou clic manuel donne `(kx_center, ky_center)` en `π/a` ;
- on stocke ces valeurs dans `session.gamma_reference` et `entry.fs_center_*` ;
- à l'affichage on translate : `kx_disp = kx − kx_center`.

Ça marche tant que `hν`, `a_lattice`, et la conversion ne changent pas.
Si l'un de ces paramètres change (autre fichier, autre run, autre
échantillon), l'angle physique de Γ reste le même mais sa position en `k`
bouge — donc la translation stockée devient fausse.

Cas concrets :
- Solaris : Γ est en général déjà au centre, problème peu visible.
- CLS : Γ doit être recentré. Aujourd'hui le tilt CLS n'a *aucun* offset
  angulaire dans `_cls_angle_to_k_pi_over_a` (ligne 328 de `arpes_io.py`) ;
  le centrage `ky` se fait uniquement par translation visuelle. Si `hν`
  change, ça décale.

### Conventions trouvées dans l'API erlab

`data.kspace.offsets` accepte 3 clés (config Type1/Type2 = Solaris DA30) :
- `delta` : offset azimut ;
- `xi`    : offset tilt ;
- `beta`  : offset de l'angle dispersif (fente).

À écrire *avant* `da.kspace.convert()`.

### Plan proposé

1. Trois angles physiques universels exposés à l'app, indépendants du lab :
   - `theta0` : offset angle dispersif (la fente).
   - `tilt0`  : offset axe lent (FS).
   - `azi0`   : offset azimutal.

2. Mapping interne par loader :
   - Solaris : `theta0→beta`, `tilt0→xi`, `azi0→delta` via `kspace.offsets`.
   - CLS : `theta0` ajouté à `angular_offset_deg` pour `kx` ; `tilt0` ajouté
     à la conversion tilt (aujourd'hui `0.0`) pour `ky` ; `azi0` ajouté à la
     rotation 2D si nécessaire.
   - Loader futur : implémenter ce contrat dans la signature `load_xxx(...,
     angle_offsets: dict | None = None)`.

3. Stockage :
   - `Session.angle_offsets: dict` (par dossier, valeur par défaut globale).
   - `FileEntry.angle_offsets: dict` (override par fichier, optionnel).

4. Conversion `(kx_center, ky_center)` → `(theta0, tilt0)` :
   ```python
   theta0 = degrees(arcsin(kx_center * pi / (a_lattice * 0.5123 * sqrt(Ekin))))
   tilt0  = degrees(arcsin(ky_center * pi / (a_lattice * 0.5123 * sqrt(Ekin))))
   ```
   On utilise `arcsin` (formule exacte cohérente avec `sin(θ)` du forward).
   Pour les petits angles ça se réduit à l'approximation linéaire ; à 20°
   l'écart commence à compter (~5%).

5. Workflow utilisateur (UI inchangée) :
   - L'utilisateur clique manuellement sur la carte ou lance "Détecter Γ"
     → `(kx_center, ky_center)` produit comme aujourd'hui.
   - On calcule `theta0/tilt0` et on les stocke dans `angle_offsets`.
   - Au chargement de tout fichier suivant du même dossier, on injecte ces
     offsets dans le loader avant la conversion → Γ tombe à `k=0` quel que
     soit `hν` ou `a_lattice`.

6. Coexistence avec `gamma_reference` actuel : le garder pour l'overlay
   ZDB / points de haute symétrie sur l'affichage, mais la *vérité* devient
   `angle_offsets`. Au moment du clic, on remplit les deux.

### Pourquoi je ne l'ai pas codé tout de suite

1. Conventions de signe `beta`/`θ_par` entre erlab et CLS pas garanties
   identiques. Risque d'écrire un bug d'inversion qui ne se voit que sur
   certains fichiers. Validation nécessaire :
   - un fichier CLS où Γ est notoirement décalé (typique Ba122 CLS) ;
   - un fichier Solaris où Γ est déjà centré (vérifier que `angle_offsets`
     vide ne touche à rien).
2. Tilt CLS aujourd'hui : aucun offset appliqué dans la conversion. Le
   centrage `ky` se fait par translation a posteriori. Migrer vers offsets
   angulaires va changer la valeur stockée de `ky_center` (en angle, pas en
   `π/a`) — il faudra interpréter les anciennes sessions comme un alias.
3. Migration `gamma_reference` → `angle_offsets` : à la première ouverture
   d'une session existante, si `angle_offsets` est vide mais
   `gamma_reference` existe, calculer une fois pour toutes les
   `theta0/tilt0` à partir de `(kx, ky)` et de `hν`, `a_lattice` du fichier
   ayant servi à la mesure (à stocker en plus dans `gamma_reference`
   actuel).

- ajout de `_center_current_bm_axis_on_gamma(gamma_bm, ref)` ;


Comportement utilisateur attendu :

  compatible, `Γ mémorisé` recentre maintenant l'axe BM ;

## Mise à jour 2026-04-29 — offsets angulaires CLS avant conversion k

Objectif :
- remplacer, quand c'est possible, la simple translation `kpar -= gamma_bm`
  par une correction injectée dans le loader CLS avant la conversion
  angle → `k`.

Principe physique utilisé :
- le loader CLS convertit l'axe dispersif avec :

```python
kx = C * sqrt(Ekin) * sin(theta - polar) * a / pi
```

- si Γ est trouvé à `kx_center`, on calcule l'angle correspondant :

```python
theta0 = degrees(arcsin(kx_center / (C * sqrt(Ekin) * a / pi)))
```

- puis le loader convertit avec :

```python
kx = C * sqrt(Ekin) * sin(theta - polar - theta0) * a / pi
```

- pour une FS CLS, même logique sur l'axe lent :

```python
ky = C * sqrt(Ekin) * sin(tilt - tilt0) * a / pi
```

Modifications dans `arpes_io.py` :
- `load_cls_txt(..., angle_offsets=None)` accepte maintenant :
  - `theta0_deg` pour l'axe dispersif/BM ;
  - `tilt0_deg` pour l'axe lent FS ;
- `load_arpes(..., angle_offsets=None)` relaie ces offsets ;
- les métadonnées indiquent :
  - `angle_offsets_applied` ;
  - `theta0_deg`, `tilt0_deg` ;
  - `kx_conversion = "theta_minus_polar_minus_theta0"` ;
  - `ky_conversion = "tilt_minus_tilt0"`.

Modifications dans `arpes_explorer.py` :
- `Session.angle_offsets` est sauvegardé dans `.arpes_session.json` ;
- après `Détecter Γ FS` ou `Viser Γ manuel`, l'application :
  - garde `session.gamma_reference` comme avant ;
  - calcule aussi `theta0_deg/tilt0_deg` depuis `(kx_center, ky_center)` ;
  - sauvegarde le résultat dans `session.angle_offsets`.
- au chargement d'une BM CLS, avant `load_arpes(...)` :
  - l'application lit `azi_ref` dans `gamma_reference` ;
  - lit `azi_bm` dans le logbook du fichier courant ;
  - projette Γ sur la direction de la BM :

```python
gamma_bm = kx_ref * cos(azi_bm - azi_ref) + ky_ref * sin(azi_bm - azi_ref)
```

  - convertit ce `gamma_bm` en `theta0_deg` spécifique à la BM ;
  - passe cet offset au loader.

Conséquence utilisateur :
- workflow attendu :
  1. charger logbook ;
  2. charger une FS CLS ;
  3. `Détecter Γ FS` ou `Viser Γ manuel` ;
  4. charger une BM du même dossier/sample ;
  5. la BM est chargée directement avec `theta0_deg`, donc Γ doit déjà
     arriver vers `k//=0`.
- dans le statut de chargement, l'application affiche maintenant :

```text
Γ offset angulaire θ0=...
```

si le loader a reçu un offset angulaire.

Filet de sécurité conservé :
- si aucun offset angulaire n'est disponible, l'ancien recentrage interface
  `kpar_display = kpar_raw - gamma_bm` reste utilisé ;
- si `angle_offsets_applied` est présent, ce recentrage post-loader est
  désactivé pour éviter une double correction.

Limites restantes :
- implémenté pour CLS/LNLS ;
- Solaris/DA30 garde pour l'instant le comportement précédent ;
- le signe de `theta0/tilt0` est celui cohérent avec la formule CLS actuelle
  `sin(angle - offset)`. À valider sur un vrai couple FS/BM : si après
  détection FS puis rechargement d'une BM le centre part dans le mauvais sens,
  le signe de l'offset devra être inversé.

## Mise à jour 2026-04-29 — sélection automatique de convention Γ FS→BM

Retour utilisateur :

Hypothèse :

  - lance `estimate_gamma_bm_mdc` sur la BM chargée ;

Comportement utilisateur attendu :

- le statut de chargement indique maintenant par exemple :

```text
Γ offset angulaire azi_minus_neg θ0=...
```

- le mot après `offset angulaire` indique la convention retenue
  automatiquement.

Limite :

## Mise à jour 2026-04-30 — géométrie CLS robuste `P/T/azi`

Motivation :
- certaines BM CLS, par exemple BM9/BM10 dans `Ba122_C05_2`, ont le même
  azimut qu'une FS de référence mais un `polar/P` très différent ;
- une projection par azimut seul ne suffit alors pas : il faut utiliser la
  géométrie manipulateur disponible dans les fichiers CLS.

Principe retenu :
- lire en priorité `P` et `T` directement dans les `*_param.txt` CLS ;
- lire `theta`, `phi`, `azi`, `hv`, etc. depuis le logbook seulement comme
  complément, car les Excel ne sont pas standardisés ;
- interpréter couramment :
  - `theta` logbook -> `polar/P` manipulateur ;
  - `phi` logbook -> `tilt/T` manipulateur ;
  - `azi` logbook -> azimut échantillon.

Modifications dans `arpes_explorer.py` :
- `FileMeta` contient maintenant aussi :
  - `polar` ;
  - `tilt`.
- `_infer_logbook_mapping(...)` tente de détecter :
  - `polar` depuis `theta`, `polar angle`, `polar deg`, `manip p`, etc. ;
  - `tilt` depuis `phi`, `tilt`, `manip t`, etc.
- `_cls_manipulator_from_param(path)` lit rapidement `P/T` dans le fichier
  CLS brut sans charger toute la donnée.
- `_cls_geometry_for_path(path, entry)` agrège la meilleure géométrie connue :
  1. `_param.txt` pour `P/T` ;
  2. session/logbook pour `azi`, `hv`, et valeurs manquantes.
- `_angle_offset_candidates_for_load(...)` ajoute des candidats de type
  `raw_polar` :

```python
theta0_target = theta0_ref + P_ref - P_target
```

  Cette convention correspond à l'idée que le centre repéré sur la FS donne
  un angle brut analyseur `theta_raw ~= P_ref + theta0_ref`, puis que la BM
  cible doit appliquer `theta_raw - P_target`.

Sélection automatique :
- les candidats historiques sont conservés :
  - `theta0`, `-theta0` ;
  - `azi_plus`, `azi_minus` et leurs signes inversés.
- les nouveaux candidats combinent aussi `azi` et `P/polar` :
  - `raw_polar` ;
  - `azi_plus_raw_polar` ;
  - `azi_minus_raw_polar`.
- le score de choix de candidat est maintenant :

```python
abs(gamma_residuel) + 0.25*MAD + 0.10*abs(k_mid)
```

  où `k_mid` est le centre de la plage `k` chargée. Cette pénalité évite de
  retenir une convention qui donne quelques pics MDC à peu près centrés mais
  laisse tout l'axe `k` très déséquilibré.

Conséquence utilisateur :
- après avoir détecté Γ sur une FS, l'utilisateur peut charger une BM avec un
  `polar/P` différent ;
- l'application teste automatiquement les conventions géométriques et affiche
  dans le statut le candidat retenu, par exemple :

```text
Γ offset angulaire raw_polar θ0=+12.967°
```

Point important :
- cette méthode ne suppose pas un Excel CLS parfaitement standard ;
- elle fonctionne d'abord avec les métadonnées brutes CLS, puis enrichit avec
  le logbook quand il apporte `azi/hv/pol` ou des champs absents.

  - `ef_kinetic_from_hv = hν - work_function` est quand même stocké ;

    ajouté ;
  - `kx = theta - P-Axis - theta0` ;
  - pour FS, `ky = P-Axis - centre_du_scan - tilt0`.

Exemple réel BESSY :
```text
Ba1220009w_Band Map B122_009.ibw
Energy Scale = Kinetic
Excitation Energy = 0
Monochromator Energy = 0
Center Energy = 43.4 eV
axe brut = 43.0 → 43.7996 eV

E-EF avec Center Energy = -0.400 → +0.400 eV
E-EF avec hν=40 eV et φ=4.031 eV = +7.031 → +7.831 eV
```

Décision :
- pour BESSY/SES, ne pas forcer `hν - φ` comme Solaris/CLS tant que l'export
  indique une référence SES locale cohérente (`Center Energy`) ;
- ne pas ignorer hν : le garder en métadonnée et avertir quand il contredit la
  référence énergie utilisée ;
- les nouveaux fichiers BESSY démarrent avec `EF offset = 0.0 eV` au lieu du
  défaut historique `+0.052 eV` ;
- les anciennes entrées BESSY non calibrées, encore au défaut exact `+0.052 eV`,
  sont ramenées à `0.0 eV` au chargement ; les fichiers déjà calibrés ou fittés
  ne sont pas écrasés.

Interface :
- le statut de chargement affiche le premier `loader_warning` quand il existe ;
- sinon il affiche la référence énergie (`refE=...`).

## Mise à jour 2026-05-01 — durcissement loader BESSY + propagation EF per-file

Contexte des fichiers Ba122 testés (`/Stage_M2/code/ba122/*.ibw`) :
- ce sont des exports SES R8000 sans `hν` valide dans la note (`Excitation
  Energy=0`, `Monochromator Energy=0`) ;
- chaque fichier a son propre `Center Energy` (ex : 36.4, 43.4, 61.4, 65.2) ;
- le logbook ancien format `Measurements Plan(Sheet1).csv` (colonnes `Num`,
  `Energy`, `Temp`, `Pol`, ...) est partiellement faux : pour les fichiers
  où Center=43.4 le logbook donne hν=41 alors que les données sont cohérentes
  avec hν≈48 ; le hν du logbook ne peut donc pas être propagé aveuglément ;
- conséquence : `Center Energy` reste la référence la plus fiable par défaut.

### 1. Détecteur BESSY strict (`arpes_io.py`)

Ancien :
```python
return b"[SES]" in note and (b"Instrument=R8000" in note or b"Energy Scale=Kinetic" in note)
```

Problème : `Energy Scale=Kinetic` est aussi dans les exports DA30 → un `.ibw`
Solaris pouvait être attrapé par BESSY.

Nouveau :
```python
return b"[SES]" in note and b"Instrument=R8000" in note
```

`Instrument=R8000-8ES202` (testé Ba122) matche `Instrument=R8000` en substring.

### 2. Mode `auto` clarifié + UI honnête (`arpes_io.py` + `arpes_explorer.py`)

`auto` reste sémantiquement = `ses_center_energy` (le plus fiable sur ces
fichiers). On ne tente PAS de basculer automatiquement sur `hν−φ` quand hν
est dispo, parce que le logbook hν est souvent erroné.

Combobox UI mis à jour :
- `BESSY Auto (Center Energy)` (au lieu de `Auto SES`) ;
- `BESSY Center Energy` ;
- `BESSY hν − φ (forcé)` (au lieu de `BESSY hν - φ`).

Tooltip explicite : indique que sur Ba122 le logbook hν est souvent
erroné, donc utiliser `hν−φ forcé` seulement après vérification.

`_bessy_energy_reference_mode()` mis à jour pour parser les nouveaux libellés
(`Auto` → `auto`, `forcé` → `hv_minus_work_function`, `Center` → `ses_center_energy`).

### 3. Support `Energy Scale=Binding` (`arpes_io.py`)

Détection :
```python
energy_scale_raw = str(ses.get("Energy Scale") or ses.get("Energy Unit") or "Kinetic")
is_binding_axis = energy_scale_raw.lower().startswith("bind") or ...
```

Si Binding :
- on ne soustrait pas `ef_kin_nominal` ;
- on flippe le signe pour respecter la convention E−EF (positif au-dessus EF) :
  `energy = -energy_raw + ef_offset` ;
- pour `kx`, on utilise `hν−φ` si dispo, sinon `Center Energy` comme proxy de Ek ;
- warnings ajoutés (`énergie SES en Binding`, `kx via proxy si hν inconnu`).

Métadonnées :
- `energy_axis_original` vaut maintenant `"binding"` ou `"kinetic"` ;
- `energy_scale_raw` ajouté pour tracer la valeur SES brute.

### 4. Fenêtre EF auto-détectée (`arpes_explorer.py::_default_ef_search_range`)

Problème observé sur Ba122 : Center Energy ≈ 43.4 mais l'expérimentateur n'a
pas centré pile sur EF → bord de Fermi à `E−EF ≈ +0.26 eV`. L'ancienne
fenêtre fixe `[-0.25, +0.05]` ratait totalement le bord.

Nouveau comportement (uniquement BESSY/ses_center_energy) :
- calcule l'EDC moyenne, prend `np.argmin(np.gradient(edc, e))` comme hint
  pour la position de la chute d'intensité (= EF) ;
- centre la fenêtre autour de ce hint : `[hint − 0.30, hint + 0.20]`,
  bornée par les limites de l'axe ;
- fallback `[-0.5, +0.5]` si la détection échoue.

Test sur 4 fichiers Ba122 : EF hint trouvé à +0.26 eV (cohérent), fenêtre
résultante `[-0.04, +0.40]` qui couvre bien le bord.

Pour les autres loaders (Solaris, CLS), comportement inchangé : `[-0.5, +0.2]`.

### 5. Propagation EF per-file (`arpes_explorer.py::_apply_ef_reference_to_current`)

Problème : un shift scalaire `Δ` mesuré sur un Au avec `Center=Cs` n'est
**pas valide** sur un fichier cible avec `Center=Ct`, car les deux axes
E−EF n'ont pas la même origine cinétique. L'utilisateur devait recalibrer
manuellement chaque fichier.

Solution :
- au moment de sauvegarder la référence (`_apply_ef_calibration_result`,
  mode scalar), on ajoute dans `ref_payload` :
  - `source_ef_kin_nominal` = `ef_kinetic_nominal` ou `ef_kinetic_from_hv`
    du fichier source (helper `_axis_zero_in_kinetic`) ;
  - `source_energy_reference` = mode énergie source.
- au moment d'appliquer (`_apply_ef_reference_to_current`, mode scalar),
  on lit `target_ef_kin_nominal` du fichier courant et on calcule :

```text
kinetic_correction = source_ef_kin_nominal - target_ef_kin_nominal
effective_shift    = base_shift + kinetic_correction
new_offset         = current_offset - effective_shift
```

Garde-fous :
- correction per-file appliquée **seulement si** source et cible ont le même
  `energy_reference` (ses_center_energy ↔ ses_center_energy, ou
  hv_minus_work_function ↔ hv_minus_work_function) ;
- sinon (mode différent ou métadonnée absente), on retombe sur le shift
  scalaire naïf et on ajoute un ⚠ explicite dans le statut ;
- `entry.ef_correction` enregistre `ref_kinetic_correction`,
  `ref_effective_shift`, `ref_source_ef_kin_nominal`, `ref_target_ef_kin_nominal`
  pour traçabilité.

Helper top-level ajouté :
```python
def _axis_zero_in_kinetic(meta) -> float | None:
    # essaie ef_kinetic_nominal puis ef_kinetic_from_hv ; renvoie None sinon
```

### Limite physique connue

La propagation per-file fonctionne **uniquement si tous les fichiers du
dossier sont au même hν réel**. Si hν varie effectivement (cas Ba122 où
certains fichiers sont à 41 eV et d'autres à 48 eV), le décalage propagé
sera faux par la différence `(hν1−hν2)`. À traiter plus tard via :
- soit calibrer un Au par hν,
- soit calculer une fonction de travail effective et propager via hν cible.

### Géométrie CLS : propagation Γ depuis une FS vers BM/FS

But initial :
- pouvoir charger une FS CLS, trouver son centre Γ, puis propager ce centre vers
  les autres fichiers du même dossier en utilisant l'azimut, sans bloquer sur la
  colonne `direction`/ZDB du logbook.

Changements :
- suppression du blocage strict quand `direction` ZDB diffère entre la FS de
  référence et la BM cible ;
- projection du Γ de référence par différence d'azimut :

```text
d_azi = radians(azi_target - azi_ref)
k_parallel = kx_ref*cos(d_azi) + ky_ref*sin(d_azi)
k_perp     = -kx_ref*sin(d_azi) + ky_ref*cos(d_azi)
```

- pour les BM CLS, `k_parallel` sert à construire l'offset angulaire appliqué
  avant conversion en `k` ;
- pour les FS CLS, les deux composantes projetées servent à convertir le centre
  Γ cible en `theta0_deg` / `tilt0_deg` ;
- les offsets CLS sont maintenant passés au loader (`angle_offsets`) avant le
  calcul de `kx`/`ky`, au lieu de corriger seulement l'affichage après coup ;
- les données non-CLS ne reçoivent plus ces offsets par erreur.

Garde-fous :
- la propagation automatique est refusée si le `polar/P` manipulateur de la BM
  diffère trop de celui de la FS de référence ;
- seuil utilisé : environ `±2°` ;
- cas observé : FS5 `P=+1.36°`, BM9 `P=-12.30°`, écart `-13.66°` → pas une
  simple rotation azimutale, donc propagation Γ refusée.

Conclusion pratique :
- les BM avec même famille de `polar/P` peuvent utiliser la FS de référence ;
- les BM trop décalées en `polar/P` doivent être regroupées avec une FS dédiée,
  ou calibrées manuellement.

- `Dossier` :

Garde-fous :

- aucun fichier n'est chargé pour calculer un groupe ;

- les valeurs manquantes restent explicitement inconnues.

À améliorer plus tard si nécessaire :
- ajouter une source de confiance plus visible (`fichier`, `_param`, `logbook`,

## ROADMAP EN COURS — refonte architecture progressive

> Cette section applique `AI_COUNCIL_PROMPT.md` à l'architecture générale de
> l'application. C'est un plan de refonte incrémentale avec avancement partiel.
> Ne pas lancer une grosse refonte globale. Chaque update doit être petite,
> testable, et validée manuellement dans l'interface avant de passer à la
> suivante. Updates A, B, C v1, C v2, D, E, F, G, H.1, H.2, H.3 et I sont
> faites. Update L (`PlotController`) est faite au niveau extraction sûre :
> les calculs et dessins Matplotlib communs sont sortis, tandis que le rendu
> détaillé MDC Fit couplé aux contrôles interactifs reste volontairement dans
> l'UI. La suite active est Update M, sauf
> décision contraire du conseil.

### Proposition soumise au conseil

Réduire les risques de bugs en sortant progressivement de `arpes_explorer.py`
les décisions applicatives et scientifiques qui ne devraient pas vivre dans la
classe UI principale :
- session / persistance ;
- logbook / matching ;
- orchestration loaders et priorité des métadonnées ;
- préparation et post-traitement des fits ;
- export résultats.

Objectif : garder l'interface PyQt comme couche d'affichage et d'interaction,
mais rendre les décisions métier testables sans lancer l'UI.

### Avis du conseil

#### Physicien ARPES
- Accord : les règles physiques critiques doivent rester explicites et
  traçables.
- Risque : une refonte qui change EF, Γ, hν, polar/tilt/azi ou la résolution
  sans trace visible est inacceptable.
- Exigence : chaque extraction doit préserver les métadonnées runtime et les
  warnings déjà visibles.

#### Expert géométrie instrumentale ARPES
- Accord : les conventions d'angles et de repères doivent rester centralisées
  ou clairement documentées.
- Risque : déplacer du code Γ/angle offsets peut inverser un signe ou changer
  l'ordre des corrections.
- Exigence : aucune modification de convention géométrique pendant les updates
  architecture. Si une convention est touchée, créer une update séparée dédiée.

#### Développeur Python/PyQt
- Accord : `arpes_explorer.py` concentre trop de responsabilités.
- Risque : bug UI indirect si on extrait sans stabiliser les signaux et l'état
  courant.
- Exigence : l'UI doit continuer à appeler des méthodes simples et ne pas
  connaître les détails des loaders/logbook/session.

#### Architecte loader / IO
- Accord : `arpes_io.py` est le socle à préserver.
- Règles non négociables :
  - l'interface ne doit jamais appeler `erlab.io.load` directement ;
  - le chargement passe par `load_arpes(...)` ;
  - `detect_format(...)` reste la porte d'entrée de détection ;
  - les métadonnées fichier valides gardent priorité sur logbook/manuel.
- Exigence : tout nouveau manager doit respecter `ARPESData` et ne pas dupliquer
  les décisions de `arpes_io.py`.

#### Red team données réelles
Cas pathologiques à garder en tête pour chaque update :
- logbook dit `hv=100` mais le header fichier dit `hv=48` ;
- fichier BESSY avec `Excitation Energy=0` mais hν logbook utile pour kz ;
- fichier CLS sans température ou sans direction ;
- `.ibw fixed cut` dont le nom ne matche pas naïvement la ligne logbook ;
- session existante avec anciens `fit_result` sans `resolution` ;
- groupe UI replié ou fichier courant masqué après refresh.

Exigence : aucun de ces cas ne doit échouer silencieusement.

#### Numéricien
- La refonte architecture ne doit pas modifier les algorithmes numériques.
- Les fonctions de fit, normalisation, résolution, EF, Γ doivent rester
  comparables avant/après extraction.
- Tout changement de résultat numérique doit être documenté comme changement
  scientifique, pas comme simple refactor.

#### UX utilisateur scientifique
- Les labels, warnings et icônes de source doivent rester visibles.
- Une extraction ne doit pas rendre le comportement plus opaque.
- Si une valeur vient du fichier, du logbook ou du manuel, l'utilisateur doit
  encore pouvoir le savoir.

### Décision finale du conseil

Décision : **implémenter une refonte progressive**, mais **refuser une refonte
globale de `arpes_explorer.py` en une seule update**.

Critère d'arrêt :
- si une étape exige de changer simultanément session + logbook + loader + fit,
  elle est trop grosse et doit être redécoupée.

Critère de validation :
- après chaque update, l'application doit se lancer ;
- chargement Solaris/BESSY/CLS doit rester possible ;
- les tests existants doivent passer ;
- une validation manuelle minimale dans l'UI doit être faite.

### État architectural actuel

- `arpes_explorer.py` est beaucoup moins chargé en logique métier, mais reste
  la couche UI principale.
- Les décisions lourdes sont maintenant séparées :
  - session : `arpes_session.py` ;
  - modèles runtime : `arpes_models.py` ;
  - logbook matching : `arpes_logbook.py` ;
  - logbook IO : `arpes_logbook_io.py` ;
  - orchestration loaders : `arpes_loader_orchestrator.py` ;
  - géométrie CLS : `arpes_cls_geometry.py` ;
  - Γ/angles : `arpes_gamma.py` ;
  - EF : `arpes_ef_controller.py` ;
  - fit : `arpes_fit_controller.py` ;
  - export : `arpes_export.py` ;
  - résolution : `arpes_resolution.py` ;
  - préparation plot : `arpes_plot_controller.py`.
- Les conventions scientifiques n'ont pas été changées pendant J/K/L.
- Les corrections automatiques restent traçables via messages, metadata ou
  session.

### Conseil recommandé pour la suite

#### Priorité 1 — Validation manuelle UI

Avant toute nouvelle extraction, faire une passe manuelle dans l'app :

1. Lancer :

```bash
python3 arpes_explorer.py
```

2. Charger un dossier avec logbook.
3. Tester au minimum :
   - chargement Solaris/BESSY/CLS ;
   - regroupements navigateur ;
   - affichage BM Raw/EDCnorm/SecDev/Curvature ;
   - correction grille BM ;
   - onglet MDC Fit ;
   - waterfall ;
   - calibration EF scalaire et poly si possible ;
   - application référence EF ;
   - FS + centrage Γ ;
   - export CSV.

Raison : les tests unitaires couvrent les décisions pures, mais pas encore les
interactions Matplotlib/PyQt ni les callbacks souris.

#### Priorité 2 — Update M, mais seulement si la validation UI est OK

Update M recommandée :
- découper `_load_file(...)` en sous-méthodes privées sans changer de module :
  - `_resolve_load_inputs(...)` ;
  - `_apply_loaded_data_to_session(...)` ;
  - `_apply_loaded_data_to_ui(...)` ;
  - `_finalize_load_status(...)`.
- découper `_build_ui(...)` seulement si nécessaire :
  - `_build_left_panel(...)` ;
  - `_build_center_tabs(...)` ;
  - `_build_right_panel(...)` ;
  - `_wire_signals(...)`.

Pourquoi M avant une nouvelle extraction scientifique :
- faible risque ;
- améliore la lisibilité ;
- prépare la maintenance sans toucher aux conventions EF/Γ/k.

#### Priorité 3 — Transmission / Phase 3

Si l'app doit être reprise par quelqu'un d'autre :
- faire `Update P` README avant packaging ;
- ensuite seulement `Update N` packaging minimal ;
- puis `Update O` CI ou hook local de tests.

Ordre conseillé :

```text
README court → pyproject minimal → test command documentée/CI
```

#### Ce que je déconseille maintenant

- Ne pas extraire `_draw_mdc_edc()` immédiatement dans un gros `PlotController`.
  Ce bloc est trop couplé aux contrôles vivants du fit MDC. Si on le sort, le
  faire comme une update dédiée avec validation visuelle avant/après.
- Ne pas mélanger Update M avec une nouvelle fonctionnalité scientifique.
- Ne pas modifier simultanément loader + normalisation + affichage : en cas de
  régression, on ne saurait plus attribuer la cause.
- Ne pas attaquer packaging complet `.app`/PyInstaller maintenant : trop tôt,
  et probablement fragile avec PyQt/scipy/matplotlib/erlab.
